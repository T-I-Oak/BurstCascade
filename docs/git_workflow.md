# Git ワークフローとバージョン管理

本ドキュメントは、Burst Cascade プロジェクトにおけるブランチ管理、バージョニング、およびリリース手順について定めます。

## 1. バージョン管理

### 1.1 バージョン表記
セマンティックバージョニング（Major.Minor.Patch）に準拠して管理します。
- バージョン更新は原則として **`package.json` の `version` フィールドのみ**を変更します。
- ゲーム内およびテストスイートには Vite 経由（`__APP_VERSION__`）で自動的に値が注入され、動的に更新されるため、HTML 等を手動で編集する必要はありません。

### 1.2 バージョン更新ルール
- **Major (メジャー)**: プロジェクトオーナーから明示的な指示があった場合、または正式リリース（V1.0）時のみ。
- **Minor (マイナー)**: **「新しい開発（新ブランチ）」を開始する際**に、現在のバージョンに 1 加える。
    - 例: 現在 `0.1.x` であれば、次の開発ブランチ開始時に `0.2.0` とする。
- **Patch (パッチ)**: コードの改修を行い、テスト担当者（ユーザー）に動作確認・検証を依頼するタイミングで 1 加える。これにより、ソースコードに変更が加わったことを明示する。
    - ※ β期間（v0.x）中もこのルールを適用し、進捗を可視化する。

## 2. ブランチ命名規則
以下のプレフィックスを基本とし、開発対象のバージョン番号（**`v[Major].[Minor]`** 形式）を含めてください。
- ブランチ名は「現在のマイナーバージョンに 1 加えたもの」を原則とします。
- 開発中にパッチバージョンが変動するため、ブランチ名にはパッチバージョンを含めないでください。
- **新規機能**: `feature/v[Major].[Minor]-[description]`
- **リファクタリング**: `refactor/v[Major].[Minor]-[description]`
- **バグ修正**: `fix/v[Major].[Minor]-[description]`

例: `feature/v0.2-data-manager`

## 3. 開発完了チェックリスト
開発を完了（マージ）する前に、以下の項目を確認してください。

1. **テスト実行**: `npm test` を実行し、全テストがパスしていること。
2. **仕様の同期**: `docs/SPECIFICATION.md` 等の関連ドキュメントが実装と同期していること。
3. **バックログの整理**: `backlog.md` の完了項目を整理すること。
4. **更新履歴の記録**: 
    - `update_history.json` を更新する。
    - ただし、**β期間（v0.x）の履歴は記録しない**（空配列 `[]` を維持）。1.0 以降から正式な記録を開始する。
5. **ユーザー承認**: 開発完了のサマリーを報告し、ユーザーから承認を得ること。

## 4. バージョン 0.x (β期間) の扱い
- 現在は正式リリース前の β期間として、`0.1.0` から再スタートしています。
- この期間中は機能の追加や大規模なリファクタリングが頻繁に行われるため、ユーザー向けの正式な更新履歴（`update_history.json`）への記録は行いません。
- 品質が安定し、ロードマップ（シェア機能実装等）が完了した段階で V1.0 とし、正式な運用を開始します。

## 5. update_history.json のデータ形式

V1.0 以降、更新履歴を記録する際は以下の JSON 構造に従ってください。

### データ形式サンプル
```json
[
  {
    "version": "1.0",
    "date": "2026-05-10",
    "title": "パフォーマンス改善",
    "description": "レンダリングエンジンの最適化により、動作がよりスムーズになりました。",
    "changes": [
      "描画処理のメモリ使用量を15%削減",
      "特定の条件下で発生していたグリッチの修正",
      "UIのレスポンス向上"
    ]
  }
]
```
